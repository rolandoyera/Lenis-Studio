# Data Model

A working reference for how we shape CRM data. The goal is canonical, reusable
shapes that compose across entities — start with the most important (the
**Client**), then layer on Projects, Invoicing, and the rest without redefining
the same ideas.

This page is part scratchpad, part reference. The **entity shapes** below
(Client, Lead), the activity **envelope**, client notes, and the client + lead
**timelines** are promoted into `lib/types.ts` and wired up; the broader event
taxonomy (proposals, invoices, the portal) is still settling here before
promotion.

## Entities

The core records everything else hangs off. Both are **org-scoped** (every doc
carries `organizationId`) and use epoch-millisecond timestamps. A Lead is the
top of the funnel; converting one **creates** a Client and links the two via
`sourceLeadId` / `convertedClientId` — neither doc is destroyed, so both keep
their own activity timeline.

### Client

A won, billable contact — either a person or a company (`isCompany`). Created
either directly or by converting a Lead. Contact and address fields are the same
set a Lead carries, so conversion is a clean copy.

```ts
interface Client {
  uid: string;
  organizationId: string;
  /** Person vs. company. Defaults false for legacy clients predating the field. */
  isCompany: boolean;
  firstName: string;
  lastName: string;
  email: string;
  phone: string;
  /** ISO 3166-1 alpha-2 code that drives phone formatting/validation. */
  phoneCountry?: string;
  company?: string;
  taxId?: string;
  taxable?: boolean;
  street?: string;
  city?: string;
  state?: string;
  zip?: string;
  country?: string;
  /** Set when this client was created by converting a Lead. */
  sourceLeadId?: string;
  createdAt: number;
}
```

### Lead

A prospect moving through the sales pipeline. Most fields are optional — leads
come in partially-filled from many sources and get enriched over time. The
qualification fields (`propertyType`, `budgetRange`, `desiredTimeline`) capture
project fit; the assignment and conversion blocks track ownership and outcome.

```ts
type LeadStage =
  | "new"
  | "contacted"
  | "qualified"
  | "proposal_sent"
  | "won"
  | "lost"
  | "on_hold";

type LeadSource =
  | "website"
  | "instagram"
  | "referral"
  | "manual"
  | "google"
  | "houzz"
  | "facebook"
  | "client"
  | "repeat_client"
  | "friend"
  | "family"
  | "other";

type PropertyType =
  | "residential"
  | "commercial"
  | "hospitality"
  | "multifamily"
  | "retail"
  | "office"
  | "other";

type BudgetRange =
  | "under_50k"
  | "50k_100k"
  | "100k_250k"
  | "250k_500k"
  | "500k_1m"
  | "over_1m";

type DesiredTimeline =
  | "asap"
  | "1_3_months"
  | "3_6_months"
  | "6_12_months"
  | "12_plus_months"
  | "not_sure";

interface Lead {

  organizationId: string;

  // Contact type

## Client Activity

Everything a client "does" or has done to their record — across projects,
proposals, invoices, the portal, files, and notes — rolls up into a single
**activity timeline** on the client. Two pieces: the **event taxonomy** (what
kinds of things happen) and the **event envelope** (the record we store for
each one). The same envelope and storage pattern power a parallel **lead**
timeline, which reuses these pieces under a narrower, sales-focused taxonomy.

### Event taxonomy

Rather than one flat union, each domain owns its event names. The client
timeline is the **composition** of those domain unions — so when Projects or
Invoicing grow their own timelines later, they reuse these exact pieces.

```ts
// Each domain owns its events…
type ClientLifecycleEvent =
  | "client_created"
  | "client_updated"
  | "client_archived"
  | "client_restored"
  | "lead_converted_to_client";

type ProjectEvent =
  | "project_created"
  | "project_status_changed"
  | "project_completed"
  | "project_archived";

type ProposalEvent =
  | "proposal_created"
  | "proposal_updated"
  | "proposal_sent"
  | "proposal_viewed"
  | "proposal_approved"
  | "proposal_declined"
  | "proposal_expired"
  | "proposal_signed"
  | "proposal_archived";

type InvoiceEvent =
  | "invoice_created"
  | "invoice_updated"
  | "invoice_sent"
  | "invoice_viewed"
  | "invoice_paid"
  | "invoice_partially_paid"
  | "invoice_overdue"
  | "invoice_voided";

type PortalEvent =
  | "portal_invite_sent"
  | "portal_invite_resent"
  | "portal_activated"
  | "portal_logged_in"
  | "portal_file_downloaded"
  | "portal_message_sent"
  | "portal_message_received";

  | "note_added"
  | "note_deleted"
  | "comment_added"
  | "email_sent"
  | "meeting_logged"
  | "call_logged";

// Leads own their own sales/relationship events…

  | "lead_created"
  | "lead_updated"

// …and the client timeline is just the union of them.
type ClientActivityType =
  | ClientLifecycleEvent
  | ProjectEvent
  | ProposalEvent
  | InvoiceEvent
  | PortalEvent
  | DocumentEvent
  | CommunicationEvent;

// The lead timeline reuses the shared comms/document pieces, kept deliberately
// narrow — pipeline movement, assignment, conversion, not every field edit.
type LeadActivityType =
  | LeadLifecycleEvent
  | CommunicationEvent
  | DocumentEvent;
```

### Event envelope

The `type` is only a label. The shape that matters is the record we persist for
every activity — **who** did it, **whose timeline** it belongs to (`source`),
**what** it refers to (`entity`), and a type-specific payload. Every record is
self-contained so it can render anywhere, including an org-wide feed.

```ts
/** Who triggered the activity. */
interface ActivityActor {
  type: "user" | "client" | "system";
  /** Optional — system-generated events have no real actor id. */
  id?: string;
  /** Denormalized for display so the timeline needs no extra lookups. */
  name: string;
}

type ActivityEntityType =
  | "lead"
  | "client"
  | "project"
  | "proposal"
  | "invoice"
  | "portal"
  | "note"
  | "file";

/** The record whose timeline this belongs to — the per-entity query key. */
interface ActivitySource {
  type: ActivityEntityType;
  id: string;
  /** Denormalized display name so feed rows render without a lookup. */
  label?: string;
}

/** What the activity points at; often equals source, sometimes a related record. */
interface ActivityEntity {
  type: ActivityEntityType;
  id: string;
  /** Denormalized label for display, e.g. "Invoice #1042". */
  label?: string;
}

/** Append-only audit record — written once, never updated or deleted. */
interface Activity {
  id: string;
  organizationId: string;
  type: ActivityType;
  actor: ActivityActor;
  /** Whose timeline this belongs to — filter on source.type + source.id. */
  source: ActivitySource;
  /** What the event is about (for top-level events, usually the source itself). */
  entity: ActivityEntity;
  /** Who can see it. Defaults to internal; opt specific events into the portal. */
  visibility: "internal" | "client_visible";
  /** Type-specific extras — kept loose for now (see "Typed payloads" below). */
  metadata?: Record<string, unknown>;
  createdAt: number;
}
```

### Example record

```ts
const example: Activity = {
  id: "act_8f2",
  organizationId: "org_lenis",
  type: "invoice_paid",
  actor: { type: "client", id: "cli_avery", name: "Jordan Avery" },
  source: { type: "client", id: "cli_avery", label: "Jordan Avery" },
  entity: { type: "invoice", id: "inv_1042", label: "Invoice #1042" },
  visibility: "client_visible",
  metadata: { amount: 4200, currency: "USD", method: "card" },
  createdAt: 1718900000000,
};
```

## Storage

Activities live in **one flat top-level collection** — not per-entity
subcollections:

```
activities/{activityId}
```

Every doc carries `organizationId` and a `source` pointer, so the same
collection answers both kinds of read:

- **Per-entity timeline** — `where organizationId == orgId && source.type == "lead" && source.id == leadId orderBy createdAt desc`.
- **Org-wide feed** — `where organizationId == orgId orderBy createdAt desc limit 20`.

These need a couple of composite indexes (Firestore prompts a one-click create
link the first time each query runs).

When an event belongs on more than one timeline, **write one doc per timeline**,
each with the appropriate `source`. **Lead → client conversion** is the
canonical case: it writes two `lead_converted_to_client` activities — one with
`source` = the lead (entity points forward at the new client) and one with
`source` = the new client (entity points back at the lead). The lead is
preserved, so its timeline survives the conversion. (A global feed can de-dupe
these at render if needed.)

> **Notes are different.** Notes stay as parent subcollections
> (`clients/{clientId}/notes`, `leads/{leadId}/notes`) — they're content owned
> by a parent and only ever read in that parent's context. Only *activities*
> were flattened.

## Conventions

Settled rules the model relies on:

- **Append-only.** Activities are immutable: create only, never update or
  delete. An edit produces a *new* event (e.g. `proposal_updated`) — it never
  mutates the prior record. The interface reflects this with a single
  `createdAt` and no `updatedAt`. This is what turns the timeline into a true
  audit log.
- **Actor name is historical.** Store `actor.name` at write time and never
  reconcile it. "Sarah Jones sent proposal ES-10004" must stay accurate even
  after she's renamed to Sarah Smith — feeds are history, not live profile
  lookups.
- **Default to internal.** New events are `visibility: "internal"` unless an
  event is explicitly opted into the client portal.
- **Update events are the norm, not the rule.** Most editable entities can emit
  update events (`proposal_updated`, `invoice_updated`). Some content types are
  intentionally stricter — notes are **create + soft-delete only** and have no
  update event.
- **Timestamps are epoch milliseconds** (`number`) everywhere, matching the
  app-wide convention in `lib/types.ts`.

## Notes

Notes contain the content; activities tell the story. A note is append-only:
**immutable after creation**, owned by its creator, and removable only via a
creator **soft-delete** — never edited, never physically deleted.

### Rules

- **No editing.** A note's body is fixed at creation. There is no
  `note_updated` event and no `updatedAt`/`updatedBy` on the document. If a note
  is incorrect, add a _new_ note rather than modifying the original.
- **Soft-delete only.** Note documents are never physically deleted. A delete
  sets `deletedAt` / `deletedBy` after a simple confirmation prompt. Only the
  note's creator may soft-delete it.
- **Visibility.** The normal notes view hides notes where `deletedAt` exists;
  audit/history views may still show them. Deleting clears clutter without
  losing the paper trail.

### Document shape

```ts
interface ClientNote {
  id: string;
  organizationId: string;
  clientId: string;
  body: string;
  createdBy: ActivityActor;
  /** Epoch milliseconds — the app-wide timestamp convention. */
  createdAt: number;
  deletedAt?: number;
  deletedBy?: ActivityActor;
}
```

Stored at `clients/{clientId}/notes/{noteId}`.

### Activity

- Creating a note → a `note_added` activity.
- Soft-deleting a note → a `note_deleted` activity.
- The activity points at the note: `entity: { type: "note", id: noteId }`.
- Activities stay append-only and immutable: soft-deleting a note never updates
  the original `note_added` record.

### Security rules (direction)

- **Create note** — any organization member.
- **Read note** — any organization member.
- **Edit note body** — never allowed.
- **Physical delete** — never allowed.
- **Soft-delete** — only the user who created the note.
- Other members may read a note but may not modify or delete it.
- Admin override is deferred and not part of v1 (see [Deferred](#deferred)).

> Notes contain the content. Activities tell the story. Notes are immutable
> after creation; only the creator may soft-delete a note; deleted notes remain
> available for audit.

## Deferred

### Typed payloads

Keep `metadata?: Record<string, unknown>` for V1 — we don't know all the
activities yet. Once proposals, invoices, projects, and the portal are actually
built in production, the real payloads reveal themselves; _then_ move to a
discriminated map so each event's extras are type-checked:

```ts
interface ActivityPayloads {
  invoice_paid: { amount: number; currency: string };
  proposal_viewed: { viewedAt: number };
}
```

Not worth solving today — it would be guessing.

### Redaction

Activities are append-only records. The only sanctioned exception is controlled
redaction.

Redaction may overwrite or tombstone specific sensitive fields, such as actor
names, metadata values, or accidentally stored PII. Redaction should be treated
as an audited administrative action, not a normal edit.

That keeps the rule clean:

> No edits, no deletes, except controlled redaction.

### Admin override

Organizations may optionally allow Admins or SuperAdmins to soft-delete notes
created by other users. This capability is deferred and not part of v1 — in v1,
only a note's creator may soft-delete it.
