# Data Model

A working reference for how we shape CRM data. The goal is canonical, reusable
shapes that compose across entities — start with the most important (the
**Client**), then layer on Projects, Invoicing, and the rest without redefining
the same ideas.

This page is part scratchpad, part reference. What's actually wired up in
`lib/types.ts` today is deliberately small: the **entity shapes** below (Client,
Lead), **client notes**, and the activity **envelope** — emitted only as a note
audit trail (`note_added` / `note_deleted`). The broader event taxonomy and
per-entity timelines (proposals, invoices, the portal, lead/client feeds) are
aspirational design — captured here but **not built**.

## Entities

The core records everything else hangs off. Both are **org-scoped** (every doc
carries `organizationId`) and use epoch-millisecond timestamps. A Lead is the
top of the funnel; converting one **creates** a Client and links the two via
`sourceLeadId` / `convertedClientId` — neither doc is destroyed, so the lead is
preserved for the record.

### Client

A won, billable contact — either a person or a company (`isCompany`). Created
either directly or by converting a Lead. Contact and address fields are the same
set a Lead carries, so conversion is a clean copy.

```ts
interface Client {
  uid: string;
  organizationId: string;
  /** Person vs. company. Defaults false for legacy clients predating the field. */
  isCompany: boolean;
  firstName: string;
  lastName: string;
  email: string;
  phone: string;
  /** ISO 3166-1 alpha-2 code that drives phone formatting/validation. */
  phoneCountry?: string;
  company?: string;
  taxId?: string;
  taxable?: boolean;
  street?: string;
  city?: string;
  state?: string;
  zip?: string;
  country?: string;
  /** Set when this client was created by converting a Lead. */
  sourceLeadId?: string;
  createdAt: number;
}
```

### Lead

A prospect moving through the sales pipeline. Most fields are optional — leads
come in partially-filled from many sources and get enriched over time. The
qualification fields (`propertyType`, `budgetRange`, `desiredTimeline`) capture
project fit; the assignment and conversion blocks track ownership and outcome.

```ts
type LeadStage =
  | "new"
  | "contacted"
  | "qualified"
  | "proposal_sent"
  | "won"
  | "lost"
  | "on_hold";

  | "houzz"
  | "facebook"

  | "repeat_client"

  | "other";

type PropertyType =
  | "residential"
  | "commercial"
  | "hospitality"
  | "multifamily"
  | "retail"
  | "office"
  | "other";

  | "under_50k"
  | "50k_100k"
  | "100k_250k"
  | "250k_500k"
  | "500k_1m"

  | "asap"
  | "1_3_months"
  | "3_6_months"
  | "6_12_months"

interface Lead {

  isCompany: boolean;
```

## Activity

Today activities do exactly one thing: give **notes** an append-only audit
trail. Creating or soft-deleting a note writes a single `note_added` /
`note_deleted` record. That's the whole of what's wired up — there are no
per-entity timelines or org-wide feed yet.

The **envelope** (further down) is the live shape. The wide event taxonomy that
follows is **aspirational** — kept as a design target for when Projects,
Proposals, Invoicing, and the portal grow their own timelines, but **none of it
is built**.

### Event taxonomy (aspirational — not built)

The design intent: rather than one flat union, each domain owns its event names,
and a timeline is the **composition** of the relevant domain unions. Captured
here so the shape is ready when those features land. The live `ActivityType` is
just the note pair shown under the envelope — not this union.


### Event envelope

The shape we actually persist for every activity — **who** did it, **whose
record** it belongs to (`source`), and **what** it refers to (`entity`). Every
record is self-contained so it carries its own context. This is the live shape;
the only `type` values written today are `note_added` and `note_deleted`.

```ts
/** Who triggered the activity. */
interface ActivityActor {
  type: "user" | "client" | "system";
  /** Optional — system-generated events have no real actor id. */
  id?: string;
  /** Denormalized for display so the timeline needs no extra lookups. */
  name: string;
}

/** The only events written today — a note's audit trail. */
type ActivityType = "note_added" | "note_deleted";

type ActivityEntityType = "client" | "note";

/** The record this activity belongs to — the per-entity query key. */
interface ActivitySource {
  type: ActivityEntityType;
  id: string;
  /** Denormalized display name so feed rows render without a lookup. */
  label?: string;
}

/** What the activity points at; often equals source, sometimes a related record. */
interface ActivityEntity {
  type: ActivityEntityType;
  id: string;
  /** Denormalized label for display, e.g. "Invoice #1042". */
  label?: string;
}

/** Append-only audit record — written once, never updated or deleted. */
interface Activity {
  id: string;
  organizationId: string;
  type: ActivityType;
  actor: ActivityActor;
  /** The record this belongs to — filter on source.type + source.id. */
  source: ActivitySource;
  /** What the event is about; for note events, the note. */
  entity: ActivityEntity;
  /** Who can see it. Defaults to internal; opt specific events into the portal. */
  visibility: "internal" | "client_visible";
  createdAt: number;
}
```

### Example record

```ts
const example: Activity = {
  id: "act_8f2",
  organizationId: "org_lenis",
  type: "note_added",
  actor: { type: "user", id: "usr_sarah", name: "Sarah Jones" },
  source: { type: "client", id: "cli_avery", label: "Jordan Avery" },
  entity: { type: "note", id: "note_19c" },
  visibility: "internal",
  createdAt: 1718900000000,
};
```

## Storage

Activities live in **one flat top-level collection** — not per-entity
subcollections:

```
activities/{activityId}
```

Every doc carries `organizationId` and a `source` pointer back at the record it
concerns. Today the only writers are note add/delete, so the collection is
effectively a note audit log — nothing reads it back yet.

The flat layout is kept on purpose: when timelines do land, the same doc can be
queried per-entity (`where organizationId == orgId && source.type == … && source.id == …`)
or org-wide (`where organizationId == orgId orderBy createdAt desc`) without
restructuring — each query would need a one-click composite index. If an event
ever needs to land on more than one timeline, the pattern is **one doc per
timeline**, each with its own `source`.

> **Notes are different.** Notes stay as parent subcollections
> (`clients/{clientId}/notes`) — they're content owned by a parent and only ever
> read in that parent's context. Only _activities_ were flattened.

## Conventions

Settled rules the model relies on:

- **Append-only.** Activities are immutable: create only, never update or
  delete. An edit produces a _new_ event (e.g. `proposal_updated`) — it never
  mutates the prior record. The interface reflects this with a single
  `createdAt` and no `updatedAt`. This is what turns the timeline into a true
  audit log.
- **Actor name is historical.** Store `actor.name` at write time and never
  reconcile it. "Sarah Jones sent proposal ES-10004" must stay accurate even
  after she's renamed to Sarah Smith — feeds are history, not live profile
  lookups.
- **Default to internal.** New events are `visibility: "internal"` unless an
  event is explicitly opted into the client portal.
- **Update events are the norm, not the rule.** Most editable entities can emit
  update events (`proposal_updated`, `invoice_updated`). Some content types are
  intentionally stricter — notes are **create + soft-delete only** and have no
  update event.
- **Timestamps are epoch milliseconds** (`number`) everywhere, matching the
  app-wide convention in `lib/types.ts`.

## Notes

Notes contain the content; activities tell the story. A note is append-only:
**immutable after creation**, owned by its creator, and removable only via a
creator **soft-delete** — never edited, never physically deleted.

### Rules

- **No editing.** A note's body is fixed at creation. There is no
  `note_updated` event and no `updatedAt`/`updatedBy` on the document. If a note
  is incorrect, add a _new_ note rather than modifying the original.
- **Soft-delete only.** Note documents are never physically deleted. A delete
  sets `deletedAt` / `deletedBy` after a simple confirmation prompt. Only the
  note's creator may soft-delete it.
- **Visibility.** The normal notes view hides notes where `deletedAt` exists;
  audit/history views may still show them. Deleting clears clutter without
  losing the paper trail.

### Document shape

```ts
interface ClientNote {
  id: string;
  organizationId: string;
  clientId: string;
  body: string;
  createdBy: ActivityActor;
  /** Epoch milliseconds — the app-wide timestamp convention. */
  createdAt: number;
  deletedAt?: number;
  deletedBy?: ActivityActor;
}
```

Stored at `clients/{clientId}/notes/{noteId}`.

### Activity

- Creating a note → a `note_added` activity.
- Soft-deleting a note → a `note_deleted` activity.
- The activity points at the note: `entity: { type: "note", id: noteId }`.
- Activities stay append-only and immutable: soft-deleting a note never updates
  the original `note_added` record.

### Security rules (direction)

- **Create note** — any organization member.
- **Read note** — any organization member.
- **Edit note body** — never allowed.
- **Physical delete** — never allowed.
- **Soft-delete** — only the user who created the note.
- Other members may read a note but may not modify or delete it.
- Admin override is deferred and not part of v1 (see [Deferred](#deferred)).

> Notes contain the content. Activities tell the story. Notes are immutable
> after creation; only the creator may soft-delete a note; deleted notes remain
> available for audit.

## Deferred

### Typed payloads

The live envelope has **no** `metadata` field — note add/delete don't need one.
When richer events land, add `metadata?: Record<string, unknown>` as the loose
escape hatch first; once proposals, invoices, projects, and the portal are
actually built in production, the real payloads reveal themselves, and _then_
move to a discriminated map so each event's extras are type-checked:

```ts
interface ActivityPayloads {
  invoice_paid: { amount: number; currency: string };
  proposal_viewed: { viewedAt: number };
}
```

Not worth solving today — it would be guessing.

### Redaction

Activities are append-only records. The only sanctioned exception is controlled
redaction.

Redaction may overwrite or tombstone specific sensitive fields, such as actor
names, metadata values, or accidentally stored PII. Redaction should be treated
as an audited administrative action, not a normal edit.

That keeps the rule clean:

> No edits, no deletes, except controlled redaction.

### Admin override

Organizations may optionally allow Admins or SuperAdmins to soft-delete notes
created by other users. This capability is deferred and not part of v1 — in v1,
only a note's creator may soft-delete it.
